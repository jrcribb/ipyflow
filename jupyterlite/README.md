# ipyflow JupyterLite demo

A self-contained [JupyterLite](https://jupyterlite.readthedocs.io) build that
runs ipyflow entirely in the browser (via Pyodide) — the target of the
"launch in JupyterLite" badge in the top-level README.

## Build it

```bash
make jupyterlite         # build ./dist
make jupyterlite-serve   # serve it (override the port with LITE_PORT=8999)
# or in one step:
make jupyterlite-dev
# then open http://127.0.0.1:8000/lab/index.html?path=demo.ipynb
```

(Equivalently: `bash jupyterlite/build.sh dist` then
`python -m http.server -d dist 8000`.)

`build.sh` builds everything from this checkout (the federated labextension, the
`ipyflow-core` wheel, and the pure-Python runtime deps bundled offline), so the
demo always reflects the current source rather than the released PyPI package.
It downloads the runtime dep wheels with `pip download`, so the **build** needs
PyPI reachable; the built site then runs fully offline in the browser.

## Deploy

`.github/workflows/deploy-jupyterlite.yml` runs `build.sh` and publishes `dist/`
to GitHub Pages. It is **manual-only** (`workflow_dispatch`) to avoid spending CI
minutes on every push — trigger it from the Actions tab (or
`gh workflow run "Deploy JupyterLite demo"`) when you want to refresh the demo.
Enable Pages → "GitHub Actions" in the repo settings first. The badge points at
`…/lab/index.html?path=demo.ipynb`.

## Contents

- `content/` — notebooks shown in the file browser (`demo.ipynb` is the landing
  notebook).
- `build.sh` — the build script (also used by CI).
- `wheels/`, `rootdist/`, `jupyter_lite_config.json` — generated by `build.sh`
  (git-ignored).

ipyflow runs in the standard **Python (Pyodide)** kernel. The labextension
silently pre-installs `ipyflow-core` on connect (install only — nothing
activates), so the user just runs `%load_ext ipyflow` to opt in, exactly as
`demo.ipynb` does. No reactive behavior appears until that `%load_ext` fires.
